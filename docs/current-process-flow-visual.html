Northstack Current Process Flow

Live at app.joinnorthstack.com. This page shows the current architecture, onboarding, and settings flows for Northstack. Last synced 2026-07-13 — see docs/current-process-flow.md for the full technical detail behind each diagram.

Where it lives

Frontend + Backend One Vercel project. Static React/Vite build plus an Express app running as a serverless function (api/index.ts).
Database Neon (serverless Postgres) via Prisma, same connection in local dev and production.
Domain + SSL app.joinnorthstack.com (Cloudflare-registered domain), SSL issued automatically by Vercel.
Deploys Every push to main auto-deploys via GitHub Actions (vercel deploy --prod).
Email Real transactional email (invitations) via Zoho Mail SMTP from no.reply@joinnorthstack.com — no third-party ESP.
Resilience Async route errors are caught and returned cleanly instead of crashing the process; Prisma retries transient connection failures.

Settings — two independent hubs

Company Settings Reached from the TopBar user menu (/company): Appearance (dark mode) and Users (roles, status, invitations) — all administrative/company-level stuff.
Settings Reached from the sidebar gear icon (/settings): Custom Fields today, structured to grow with more categories later.
No cross-navigation Neither hub links to the other — this was a deliberate correction after an earlier round merged them into one shared page.
Ownership is unique Promoting someone to owner is an atomic transfer (the acting owner is demoted to admin in the same transaction) — a tenant can never have zero or multiple owners.

HR & Clients modules

Parity Employees and Clients share the same pattern: dynamic custom fields, client-side search, inline edit.
Generic custom fields One CustomFieldValue model (tenantId + entityType + entityId) — no schema change needed when a future module (e.g. Payments) is added.
Employee self-access An Employee can optionally link to a User via invite, giving them their own login.
Status enums differ Client statuses (prospect/active/inactive/archived) are not the same set as Employee statuses (active/inactive/pending).
