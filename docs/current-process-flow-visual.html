Northstack Current Process Flow

Live at app.joinnorthstack.com. This page shows the current architecture, onboarding, settings, and PTO flows for Northstack. Last synced 2026-07-14 — see docs/current-process-flow.md for the full technical detail behind each diagram, and docs/database-schema.md for the data model.

Where it lives

Frontend + Backend One Vercel project. Static React/Vite build plus an Express app running as a serverless function (api/index.ts).
Database Neon (serverless Postgres) via Prisma, same connection in local dev and production.
Domain + SSL app.joinnorthstack.com (Cloudflare-registered domain), SSL issued automatically by Vercel.
Deploys Two branches, two pipelines: push to main deploys the app, push to landing deploys the marketing site — each via its own GitHub Actions job (vercel deploy --prod).
Email Real transactional email (invitations) via Zoho Mail SMTP from no.reply@joinnorthstack.com — no third-party ESP.
Resilience Async route errors are caught and returned cleanly instead of crashing the process; Prisma retries transient connection failures.

Settings — two independent hubs

Company Settings Reached from the TopBar user menu (/company): Appearance (dark mode) and Users (roles, status, invitations) — all administrative/company-level stuff.
Settings Reached from the sidebar gear icon (/settings): Custom Fields, Statuses, and PTO Policies — grew from 1 category to 3 without any structural rework.
No cross-navigation Neither hub links to the other — this was a deliberate correction after an earlier round merged them into one shared page.
Ownership is unique Promoting someone to owner is an atomic transfer (the acting owner is demoted to admin in the same transaction) — a tenant can never have zero or multiple owners.
flowchart TD A[TopBar user menu] -->|Company Settings| B["route: company"] B --> C[Appearance] B --> D[Users: roles, status, invitations] E[Sidebar gear icon] -->|Settings| F["route: settings"] F --> G[Custom Fields] F --> H[Statuses] F --> I[PTO Policies]

HR & Clients modules

Parity Employees and Clients share the same pattern: dynamic custom fields, client-side search, inline edit.
Generic custom fields One CustomFieldValue model (tenantId + entityType + entityId) — no schema change needed when a future module (e.g. Payments) is added.
Employee self-access An Employee can optionally link to a User via invite, giving them their own login.
Statuses are configurable now No more fixed enums — StatusDefinition is a per-tenant, per-module catalog (name, color, order, default, active). Every status change is logged in StatusHistoryEntry as a name snapshot, not a live FK.
Employee hierarchy Employee.managerId ("reports to"), self-referential, with cycle detection. Every tenant owner gets an auto-created Employee record so they always appear as a manager option.

PTO / vacation system

Configurable policies PtoPolicyDefinition per tenant — name, color, accrual (fixed annual or monthly), days/year, paid, requires approval.
Per-employee assignment Policies aren't tenant-wide by default — assigned per employee via EmployeePtoPolicy.
Hierarchy-routed approval A request's approver snapshots the employee's manager at request time; owner/admin can always override. No-approval-needed policies auto-approve instantly.
Balance, derived No balance table — computed on the fly from assignment date, accrual settings, and approved/pending requests for the current calendar year.
Calendar on Overview /overview is now the home screen after login, showing a month calendar of who's on approved or pending PTO — open to every tenant member, not just admins.
