- 2026-07-08 (más tarde): al probar el registro en el navegador se detectaron 3 problemas de UX/validación. Se reemplazaron los 7 `alert()` del navegador (Login, Register, Dashboard) por mensajes inline reutilizando `.alert-error` (ya existía en `App.css` sin usar) y una nueva clase `.field-error` para los mensajes por campo.

- 2026-07-10 (más tarde): el formulario de editar empleado no mostraba los custom fields para completarlos. Causa: el backend solo podía *crear* valores, no actualizarlos — reeditar hubiera generado duplicados. Se agregó `updateCustomFieldValue`/`findCustomFieldValueById` en `customFieldService.ts` y el endpoint `PATCH /api/hr/employees/:employeeId/custom-fields/:valueId` (con verificación de que el valor pertenece al empleado, mismo patrón de tenant ownership). Frontend: al abrir "Edit" se precargan los valores existentes (`editCustomFieldValues`/`editCustomFieldValueIds`), se agregan los mismos inputs dinámicos que en el alta, y al guardar se actualiza el valor existente o se crea uno nuevo según corresponda. Verificado con `curl`: crear valor → actualizar el mismo valor → confirmar que sigue habiendo un solo registro (no duplicó). De paso se encontró que el backend crashea completo si Neon tarda en responder (se cayó el proceso una vez durante la prueba) — anotado como pendiente de infraestructura, no corregido en esta ronda.
- 2026-07-10 (más tarde aún): a pedido del usuario, se rediseñó `CustomFieldValue` de FKs por módulo (`employeeId`/`clientId` opcionales) a modelo genérico: `tenantId` (directo, no derivado por join) + `entityType` (enum ya existente) + `entityId` (sin FK real, validado en código). Motivo: con FKs por módulo, cada módulo nuevo (Payments, etc.) necesita otra columna en la tabla — con el modelo genérico nunca más hace falta tocar el schema al agregar un módulo. `customFieldService.ts` reescrito con funciones genéricas (`listCustomFieldValuesForEntity`, `listCustomFieldValuesForEntities` para el caso bulk de `listEmployees`). `employeeService.ts` `listEmployees` ya no usa `include` de Prisma (no hay relación real) — arma `customFieldVals` a mano con un segundo query, manteniendo el mismo contrato de respuesta que ya consumía el frontend (cero cambios en frontend). Antes del `prisma db push` se hizo `DELETE FROM "CustomFieldValue"` dirigido (no `--force-reset`, que hubiera borrado toda la base) para no perder tenants/usuarios/empleados — solo se perdieron los 5 valores de custom fields de prueba, como se había acordado. Verificado end-to-end con `curl`: datos previos (tenant, empleado) sobrevivieron intactos, custom fields se pueden crear/actualizar sin duplicar, y la protección cross-tenant (IDOR) sigue funcionando con el nuevo modelo.
